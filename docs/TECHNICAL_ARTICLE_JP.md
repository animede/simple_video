# かんたん動画 Standalone 技術解説記事

この文書は、`simple_video_app` を「なぜ使いやすく、何が技術的に成立しているか」を解説するための詳細版ドキュメントです。

対象読者:
- 運用担当者（導入・障害切り分け）
- 開発者（機能追加・保守）
- 技術記事として背景を説明したい執筆者

---

## 1. アプリの到達点

このアプリは、次の制作工程を1つの画面で完結させることを目的に設計されています。

- 画像生成（T2I）
- 画像Edit（I2I）
- 動画生成（T2V / I2V / FLF）
- 1分を超える長尺動画の生成
- 音楽生成（T2A）
- MV作成（M2V: 音楽に動画を重ねる）
- PV作成（V2M: 動画に音楽を重ねる）

ポイントは「単機能の寄せ集め」ではなく、制作フローを連結したことです。
ユーザーは素材作成→調整→合成までを同一UIで進められます。

---

## 2. システム全体像

`simple_video_app` は以下の3層で構成されています。

1. UI層（ブラウザ）
   - `static/index.html`
   - `static/js/simple_video.js`
   - `static/js/bootstrap.js`

2. API層（FastAPI）
   - `app.py`
   - ジョブ投入、進捗監視、出力配信、ファイル一覧/削除、Help配信

3. 推論層（ComfyUI）
   - `/prompt` `/history` `/progress` を利用
   - workflow JSON を使って画像・動画・音楽を生成

この分離により、UI側は体験設計に集中し、生成実体はComfyUIに委譲できます。

---

## 3. なぜ「作りやすいアプリ」になったのか

### 3.1 制作パイプラインを分断しない設計

一般的な生成ツール運用では、
- 画像生成ツール
- 動画生成ツール
- 音楽生成ツール
- 動画編集ツール

を跨ぐため、素材の受け渡しで手戻りが起きます。

本アプリでは以下をアプリ内部で連結しています。

- T2Aで作った音源をM2Vへ直結
- T2V/I2Vで作った動画をV2Mへ直結
- `video_audio_merge` による最終合成をワンクリック化
- Filesブラウザで成果物を即確認

### 3.2 「音声付き動画」を明確に識別

成果物の混在は、運用上の混乱を生みます。
そこで本アプリでは:

- `output/movie` を音声付き動画の格納先として運用
- Filesに `Music Video` フィルタを実装
- 一覧上に `🎵音声付き` バッジを表示

これにより、ユーザーは「無音動画」と「音楽付き動画」を明確に区別できます。

### 3.3 独立アプリとしてのデータ境界

Files一覧APIは `simple_video_app/output` 配下を対象にし、
アプリ内成果物を優先的に扱います。

これにより、
- 何がこのアプリの生成物か
- どこを見れば成果物が揃っているか

が明確になります。

## 4. 主要機能の技術フロー

## 4.1 画像生成（T2I）

1. ユーザーが「何を描きたい？」を入力
2. `generate` APIへ workflow + prompt を送信
3. ComfyUIがT2I workflowを実行
4. 出力を `output/image` に反映
5. UIがプレビュー更新

## 4.2 画像Edit（I2I）

1. 参照画像（key/ref）を指定
2. 編集プロンプトを入力
3. I2I workflow実行（必要に応じて複数画像参照）
4. 生成画像を次工程（動画化）へ連結

## 4.3 動画生成（T2V / I2V / FLF / 混在トランジション）

- T2V: シナリオやシーンプロンプトから動画生成
- I2V: 画像を起点に動画化
- FLF: 最初/最後フレームの整合性を保つ連続化
- **混在トランジション** (`mixedTransitions: true`): LLM がシーン間の関係を判定し、FLF / I2V / カット等を境界ごとに自動割り当て。`renderTransitionEditor()` でユーザーが事後調整可能。

生成結果は主に `output/video` へ格納され、再結合（concat）にも利用されます。

## 4.4 音楽生成（T2A）

- タグ、歌詞、テンポ等を入力
- ACE-Step workflowで音源生成
- 出力は `output/audio`

## 4.5 M2V / V2M（音声付き動画の生成）

### M2V
- 音楽（既存/生成）を基準に動画を作成
- 最終段で `video_audio_merge`
- 合成結果を `output/movie` に保存

### V2M
- 動画尺に合わせて音楽を生成
- `video_audio_merge` で音声付き動画を生成
- 結果は `output/movie`

「最終成果物を movie に寄せる」ことで、運用上の視認性が高まります。

---

## 5. Filesブラウザ設計

Filesブラウザは単なる一覧ではなく、運用向けに次の要件を満たします。

- 種別フィルタ（All / Image / Video / Music Video / Audio）
- ソート（日時・名前・サイズ）
- 複数選択削除
- サムネイルのモーダル再生
  - 画像: 拡大表示
  - 動画: モーダル再生
  - 音声: モーダル再生

これにより「作って終わり」ではなく、成果物の選別・確認・整理までを同一画面で行えます。

---

## 6. モデル/ワークフロー管理方針

`README.md` ではモデルを以下の役割で明示しています。

- UNET
- GGUF UNET
- VAE
- CLIP / Text Encoder
- LoRA
- Checkpoint

この整理は「どこに置くべきか」を明確にし、導入障害を減らします。

また、workflowに記載された正式ファイル名を基準に運用するため、
配布名差異がある場合は workflow 側を環境に合わせる必要があります。

---

## 7. 運用時のトラブルシュート視点

### 7.1 `ComfyUI /prompt failed`

- `node_errors` で失敗ノードを特定
- 入力画像不足、モデル名不一致、workflow不整合を優先確認

### 7.2 出力が見えない

- Filesのフィルタ条件を確認
- `movie` は音声付き動画として管理
- APIレスポンスで `path` を確認し、UI表示と切り分け

### 7.3 M2V/V2Mの最終成果物確認

- 最終合成結果は `output/movie` を確認
- Filesでは `Music Video` フィルタを使う

---

## 8. 拡張時のガイドライン

機能追加時は次の原則を守ると品質が落ちにくくなります。

1. 生成物の保存先を明示する
   - 例: 最終成果物は `movie`、中間素材は `video` / `audio`

2. UI上で識別可能にする
   - フィルタ、バッジ、ラベルを付与

3. API契約を先に固める
   - 返却項目（path/subfolder/media_type）を安定化

4. ワークフロー差し替え可能性を維持する
   - workflow ID aliasを経由

---

## 9. まとめ

`simple_video_app` は、

- 画像生成
- 画像編集
- 動画生成
- 音楽生成
- 音声付き動画合成（MV/PV）

を1つの制作体験として統合したStandaloneアプリです。

特に、`movie` 管理と Files上の識別改善により、
「何が最終成果物か」をユーザーが迷わず判断できるようになりました。

この点が、実運用で“使えるアプリ”になった最大の要因です。
